# 价值定位与审计归因 —— 对「只读组织下本项目是否还有价值」这条反馈的回应

> 背景:有人提了一条反馈,质疑在一个「完美只读组织」里本项目是否还有价值。
> 本文记录我们对这条反馈的完整分析、该承认的地方,以及由此推导出的产品定位与
> 审计(oid → log)必须补齐的功能。这也是 `oid-log-tracking` 这个方向存在的动机。

---

## 0. 反馈原文(转述)

用户组织可能有两种情况:

1. **混合权限(场景一)**:部分工程师对某些资源有 Contributor,部分只有 Read-only。
   → 本项目价值明确:强制 AI 只能在只读 scope 里操作。

2. **完美只读(场景二)**:所有工程师只有 Read-only,所有写权限都由 Service Principal /
   CI/CD(部署 pipeline)承担。
   → 质疑:这时本项目是不是就没价值了?因为
   - 用户理论上可以直接在本地用 Claude Code 跑 az;
   - 本地跑,Azure 日志里显示的是用户本人;走本项目,日志里显示的是 SP,**丢了归因**。

---

## 1. 结论

**这条反馈只在一个很窄的点上成立,但它整个论证建立在一个错误的框架上 —— 它把本产品
理解成「把 AI 关进只读牢笼的工具」。**

如果产品的本质真的只是「强制只读」,那么场景二确实会让它失去大部分价值。但本项目的
架构(两个工具 `diagnose_bash`→Reader SP / `action_bash`→Contributor SP、按 group 路由、
OBO 门禁、沙箱执行)说明它真正在做的是另一件事:

> **把「人的常驻身份」和「执行身份」解耦,并让这条边界策略化、可审计、最小权限。**

在这个框架下,场景二不但没有杀死产品,反而暴露了反馈自身的三个隐藏前提 —— 每一个都
站不住。

---

## 2. 场景二的三个隐藏前提,逐一拆解

### 前提 1:「所有写操作都走 CI/CD」—— 漏掉了一整类操作

CI/CD 覆盖的是**声明式部署**(infra-as-code、应用发布)。但真实运维里有一大类
**交互式的 day-2 / break-glass 操作**根本不属于 pipeline:

- 重启卡死的服务 / 容器、临时 scale up 扛流量
- 事故中 kill 掉跑飞的查询、清空堆积的队列、drain 一个节点
- 重跑失败的 job、临时改一个 feature flag

这些是**写操作**,但不是「部署」。凌晨两点出事时没人会去改 Terraform 再走一遍 pipeline。
「完美组织」今天靠什么做这些?无非三条路:

- (a) 有人其实握着常驻 Contributor → 「完美只读」的前提当场破产;
- (b) PIM 临时提权 + 本地 CLI;
- (c) 手动触发 runbook。

**本项目的 `action_bash` + Contributor SP + group 门禁 + 审计,恰好是这条路的最优解:
一个受治理的、即时(JIT)的、留痕的写通道。** 人没有任何常驻写权限,写全部经过一个
策略化的收窄入口。这比「PIM 提权后在本地裸跑 az」更好治理、更可审计。

→ **场景二不但没杀死产品,反而暴露了它自己的盲区。**

### 前提 2:「人是只读所以就安全了」—— 控制面只读 ≠ 数据面只读

这是最技术、最有杀伤力的一点。一个人「只读」取决于他手里的 role,而**很多偏读的 role
带着 `listKeys` / `listCredentials` 这类 action,它们是控制面的读、数据面的写**:

| 命令 | 表面 | 实际后果 |
|---|---|---|
| `az storage account keys list` | 「读一下」 | 拿到账户 key,数据面全读写 |
| `az cosmosdb keys list` | 「读一下」 | 拿到读写连接串 |
| `az acr credential show` | 「读一下」 | 拿到 registry 凭据 |
| `az servicebus namespace authorization-rule keys list` | 「读一下」 | 拿到可发消息的 key |
| `az webapp deployment list-publishing-credentials` | 「读一下」 | 拿到发布凭据 |

所以「完美只读组织」在数据面上往往根本没锁死。而本项目可以给 Reader SP 配一个
**真正最小、不含 listKeys 的 role**,再叠加命令白名单门禁 ——
**SP 可以比人本身的「只读」权限更小。**

→ 「AI 不该看到你能看到的一切」在这里是能落地、能演示的卖点。

### 前提 3:「本地直接用 Claude Code 跑 az 就行」—— 无视了 AI 的爆炸半径

「本地跑」= 让一个 LLM 拿着**工程师本人的完整常驻身份、在他的笔记本上、用他磁盘上的
token** 执行命令。一次 prompt injection(读了一个恶意 issue、一份被投毒的日志)就能:

- 用他的 token 做他能做的一切;
- 把租户级只读能读到的东西全部 exfil;
- 叠加前提 2,直接 exfil 出可写的 key。

没有沙箱、没有命令门禁、没有集中审计,凭据还落盘。

**本项目:窄身份 + 沙箱(microVM,不碰笔记本)+ 命令门禁 + 集中审计 + ACA 路径下凭据
从不落地(passwordless FIC)。** 这层价值在**场景一和场景二里同时成立**,而且和
「人自己是读是写」完全正交 —— 反馈完全没触及这一层。

---

## 3. 关于「日志显示 SP、丢了归因」—— 要反过来看,但前提是审计做扎实

反馈说:本地跑 Azure 日志显示真实用户,走本项目就显示 SP,丢了归因。
**对写路径,这个逻辑其实是反的:**

- 人有常驻 Contributor 本地跑 → 日志里是本人,**但代价是他一直握着大范围写权限**
  (正是组织想消除的风险);
- 走 MCP → 人**零常驻写权限**,写经过窄 SP + 策略入口,应用层记录 user→action 映射。

Azure 原生日志里那个 SP **不是 bug,是「这次写被策略中介过」的签名**;真实用户靠 MCP
自己的审计记录关联回去。对安全敏感的组织,「受治理的收窄入口 + 应用层归因 + 最小常驻
权限」通常比「原生日志好看」更值钱。

### ⚠️ 但这个反驳只有在应用层审计足够强时才成立

当前代码里,归因只有 `src/mcp-server/main.py` 中间件的一行:

```python
logger.info("tool call by user_oid=%s tool=%s", oid, context.message.name)
```

有 who 和 tool,但**缺少**:具体命令、目标资源、执行结果、以及能关联回 Azure Activity
Log 的 correlation id。**如果不把这条做成一等公民,反馈这一枪就打中了。** 这是产品必须
补的功能,不是可以靠嘴辩过去的点 —— 也正是本目录(`oid-log-tracking`)要解决的核心。

---

## 4. 该承认的地方(不自我安慰)

- 如果一个组织**真的**满足:没有任何交互式写需求(连 break-glass 都走 pipeline)+ 人手里
  的「只读」确实数据面也干净 + 不担心 AI/注入的爆炸半径 + 不需要一块集中的「AI 到底干了
  啥」审计面板 —— 那么对**纯只读**用法,本项目确实退化成一层较薄的封装。这种组织存在
  (小团队、高信任、无合规要求)。**没有产品适配所有组织,这没关系。**
- 纯只读的卖点(沙箱 + AI 权限收窄 + 集中审计)比写路径的卖点弱、更难卖。这点反馈说得对。

---

## 5. 由此推导的战略定位与待办

1. **别接受「只读牢笼」这个定位。** 把叙事重心搬到写路径:**面向交互式 day-2 运维的、
   受治理的即时写(governed JIT write)**。连「完美组织」也需要它,而它们今天只能用
   PIM + 本地 CLI 笨拙地凑。这是场景二里的真正立足点。
2. **把审计做成一等公民(本目录的核心工作项):**
   `user oid ↔ SP action ↔ 具体命令 ↔ 目标资源 ↔ 结果 ↔ 能回连 Azure Activity Log 的
   correlation id`。这一步直接中和「归因」那个反对意见。
3. **主打 AI 特有风险:** prompt injection 爆炸半径、笔记本凭据暴露、agent 权限收窄。
   这层价值两个场景通吃。
4. **让 Reader SP 严格小于人的「只读」**(去掉 listKeys 类 action + 命令白名单),把
   「AI 不该看到你能看到的一切」变成可演示的卖点。

---

## 6. 一句话总结

反馈发现了**营销定位**的弱点(我们把自己说成了 read-only 工具),但没发现**架构**的真正
价值(把人的常驻身份和执行身份解耦,并让这条边界策略化、可审计)。真实世界里的组织,
绝大多数是场景一,或者是「自以为是场景二、其实藏着写需求和数据面漏洞」的场景二。
而要让这个价值真正立得住,**oid → 具体操作 → Azure log 的可关联审计链**必须补齐 ——
这就是本方向存在的理由。
