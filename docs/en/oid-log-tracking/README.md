# User Attribution for SP Operations: Tracing Every Shared Service Principal Tool Call Back to a Real Person (Document Index)

This folder contains the complete **design and implementation trajectory** of **Identity-Aware MCP's "User Attribution / OID Tracking"** — from "why the shared SP model loses attribution" to "using a correlation ID on a real ACA to link the MCP audit table with native Azure logs, and consolidating it into a formal deployment runbook" (corresponding to the `oid-log` branch, not yet merged into `main`). Core implementation: `src/mcp-server/audit.py` + `main.py` (middleware/`_exec`) + `executor.py` / `worker.py` / `sandbox_manager.py` UA injection + `provisioning/aca/modules/{mcp-observability,environment,rbac,storage,mcp-app}.bicep` (feature commit `08d2f10`, retired standalone `534dd09`).

**One-sentence solution:** Authoritative attribution is written by the MCP server itself to a Log Analytics table (Layer 1). In native logs, only one "key" is left — a correlation GUID injected into the User-Agent (Layer 2). Any service that logs UA can use this key to join back to the authoritative table, transforming logs that only show "shared SP + sandbox IP" into "which real person, which IP, which call."

**Read in the order of the table below** for the logical sequence (also roughly the creation order): Clarify the value gap (#1) → Survey the Azure logging system and make a selection (#2) → Provide code and provisioning specifications (#3) → Deploy to ACA and perform three-point closed-loop verification (#4) → Consolidate into a reusable formal deployment runbook (#5). If you just want to **deploy/verify**, jump directly to **#5**; if you want to understand **why it's designed this way**, start from **#1**.
