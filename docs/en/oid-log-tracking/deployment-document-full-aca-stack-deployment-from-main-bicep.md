## 0. Two Deployment Scenarios (Distinguish First)

| Scenario | Prerequisite | Covered by This Document |
|---|---|---|
| **Converge Deployment** | Stack is already running; want `main.bicep` to be the single source of truth / deploy a new image | ✅ Main path (§3–§7) |

The current production environment is a **converge** scenario. **Unless stated otherwise, the rest of this document refers to converge deployment.**

## 2. Mandatory Parameters ★ (Default Values are **Wrong** for Production)

The default values in `main.bicep` are intended for a "brand new example stack". Using them directly will break production. **The following 6 parameters must be explicitly passed:**

| Parameter | Value to Pass | Source | Consequence of Not Passing / Using Default |
|---|---|---|---|
| `name` | `dataops-aca` | Fixed prefix | Default `dataops-mcp` → **Creates a separate parallel stack** |
| `resourceGroupName` | `dataops-aca-rg` | `.env.aca` `ACA_RESOURCE_GROUP` | Default `${name}-aca-rg`=`dataops-aca-aca-rg` → **Wrong RG** |
| `location` | `westus2` | `.env.aca` `ACA_REGION` | —— |
| `mcpImage` | `…azurecr.io/mcp-server:<tag>` | Current image of the running app | Default public placeholder image → **Reverts MCP to placeholder image** |
| `sandboxImage` | `…azurecr.io/mcp-sandbox:latest` | Running `SANDBOX_DISK_IMAGE` | Default empty → sandbox falls back to public ubuntu disk |
| `mcpClientSecret` | (Cloud OBO secret, **retrieve now**) | Container app secret, see §4 | Default empty → bicep injects placeholder → **Breaks OBO login** |

> Current image tag lookup: `az containerapp show -g dataops-aca-rg -n dataops-aca-mcp --query "properties.template.containers[0].image" -o tsv`

## 3. Secrets Topic: Why `mcpClientSecret` Must Be the Real Value ★

**Common Misconception:** "Isn't the cloud using FIC now? Why do we still need a secret?" — This confuses two different credentials:

| Credential | App | Purpose | Replaced by FIC? |
|---|---|---|---|
| Worker SP secret (diagnose/action) | `d09dfd39` / `c58c15ae` | Worker obtains Azure tokens to execute az | ✅ **Yes** (FIC: sandbox MI → worker SP) |
| **MCP OBO secret** (`mcp-client-secret`) | MCP app `88de6a37` | MCP server **uses OBO to exchange tokens and call Graph to check groups** | ❌ **No** |

Evidence: MCP app `88de6a37` has **no federated credentials**, only 1 client secret; code
`MCP_CLIENT_SECRET = os.environ["MCP_CLIENT_SECRET"]` (hard dependency), each tool call uses `acquire_token_on_behalf_of`.

- **The `MCP_CLIENT_SECRET` in the root `.env` file is for the local app `ec0442d1`, not the cloud app. Do not use it.**
- The cloud secret exists only in the container app secret (write-only), **retrieve it now** (value is not stored on disk or printed):
  ```bash
  MCP_SECRET="$(az containerapp secret show -g dataops-aca-rg -n dataops-aca-mcp \
    --secret-name mcp-client-secret --query value -o tsv)"
  ```
- Consequence of passing empty / wrong value: `AADSTS7000215 invalid client secret` → group membership resolution fails → **both tools are not exposed**.

## 4. Full Deployment Command (Converge) ★

```bash
cd provisioning/aca

# (Optional but recommended) Pre-check blast radius, see §5
# az deployment sub what-if --name pre-check --location westus2 -f main.bicep --parameters ...(same as below)

# 1) Retrieve cloud OBO secret now (same shell, value not printed)
MCP_SECRET="$(az containerapp secret show -g dataops-aca-rg -n dataops-aca-mcp \
  --secret-name mcp-client-secret --query value -o tsv)"
[ -z "$MCP_SECRET" ] && { echo "ABORT: secret fetch failed"; exit 1; }

# 2) Converge entire stack from main.bicep (idempotent)
az deployment sub create \
  --name main-converge --location westus2 \
  --template-file main.bicep \
  --parameters \
      name=dataops-aca \
      resourceGroupName=dataops-aca-rg \
      location=westus2 \
      mcpImage=dataopsacaacrvyq3trlvkn4za.azurecr.io/mcp-server:<TAG> \
      sandboxImage=dataopsacaacrvyq3trlvkn4za.azurecr.io/mcp-sandbox:latest \
      mcpClientSecret="$MCP_SECRET"
```

> **No need to manually add registries anymore.** The old version of `mcp-app.bicep` did not declare `configuration.registries`, so deployment would remove it, requiring a post-deployment `az containerapp registry set` to fix. This is now fixed in the bicep (§7); `azurecr.io` images will automatically declare the registry.

`--name` should be a new unused name each time (sub deployment names are occupied; `main` is already taken historically, do not use it).

## 8. Cold Deployment Differences (New Environment, Appendix)

In a cold deployment, the AcrPull role is only available after the app is created and the `rbac` module completes, but the private image needs to be pulled during app creation →
**Bootstrap with a public placeholder image first, then switch to the private image:**

1. `az deployment sub create ... mcpImage=<default public placeholder> mcpClientSecret=''`
   — At this point `useAcrRegistry=false` → no registry declared, app starts with public image, `rbac` grants AcrPull.
2. `az acr build` / push the real `mcp-server` + `mcp-sandbox` images to ACR.
3. Re-run `az deployment sub create ... mcpImage=<ACR image> mcpClientSecret=<real value after reset>`
   — At this point `useAcrRegistry=true` → registry automatically declared, AcrPull is ready, private image pull succeeds.
4. Run `provisioning/aca/write-env.sh <deployment-name>` to generate `.env.aca`; follow prompts to add user to AD group,
   `az ad app update --identifier-uris api://<appId>`, `az containerapp secret set` to inject OBO secret.

Converge deployment (§4) is directly the "step 3" steady state.

## 9. Rollback

Deployment is **incrementally idempotent**; failure does not delete resources. If a deployment causes an abnormal MCP app revision:

```bash
az containerapp revision list -g dataops-aca-rg -n dataops-aca-mcp \
  --query "[].{name:name, active:properties.active, healthy:properties.healthState}" -o table
az containerapp revision activate -g dataops-aca-rg -n dataops-aca-mcp --revision <previous healthy revision>
```

If the OBO secret is accidentally overwritten: `az containerapp secret set -g dataops-aca-rg -n dataops-aca-mcp \
  --secrets mcp-client-secret=<real value>` (real value must be retrieved from elsewhere or regenerated via `az ad app credential reset`).
