# 分布式原语 · 数据库 · 编排:同一套并发控制的三个展厅

你在读 [MCP-水平扩展-分布式锁与Reaper选主.md](MCP-水平扩展-分布式锁与Reaper选主.md) 时问了一句很关键的话:

> *"这一套分布式锁、乐观锁、reaper,还有主从、主主……听起来跟 PostgreSQL 也是一样的吧?"*

**是的,而且这是个值得点破的认知飞跃。** 这些东西**既不是 Redis 发明的,也不是数据库发明的** —— 它们是更底层的**并发控制(concurrency control)+ 分布式系统(distributed systems)**理论。你觉得"像在写数据库",是因为**数据库正是大多数人第一次遇到这些概念的展厅**。

这篇把三个展厅并排摆开:

- **通用原语**:概念本身(锁、复制、选主、对账……)。
- **PostgreSQL**:它在关系型数据库里长什么样(你大概率已经熟)。
- **本项目(Redis + ARM sandbox)**:它在这套架构里长什么样。

读完你能回答:哪些是"换个系统名字都一样"的通用真理;哪些地方三者**真的分道扬镳**,以及为什么本项目的骨架**一半像数据库、一半像 Kubernetes**。

前置:[MCP-水平扩展-分布式锁与Reaper选主.md](MCP-水平扩展-分布式锁与Reaper选主.md)、[Redis-基本语法入门.md](Redis-基本语法入门.md)。

---

## TL;DR

- **锁、乐观/悲观、主从/主主、选主** 是**通用并发控制原语**,PostgreSQL 有近乎 1:1 的对应物 —— 所以"像数据库"是**真像**。
- 但有**三点关键分野**,决定了本项目**不只是"跑在数据库上的 CRUD"**:
  1. **锁的释放模型不同**:Redis 靠 **TTL / 租约**,Postgres advisory lock 靠**连接存活**。
  2. **被保护的是"外部、不可回滚的副作用"(建 sandbox),不是一行数据** —— ACID 到数据库边界就失效,只能退到**尽力而为的锁 + 幂等 + 事后对账(reaper)**。这是 **saga / 最终一致性**的世界,不是 ACID 的世界。
  3. **Redis 在这里就是被当协调数据库用的**,你完全**可以换成 Postgres**;选 Redis 是"选对工具",不是"能力差距"。
- 一句话归类:**锁 / 主从 = 数据库那一套;reaper / "建 sandbox 不可回滚" = 编排(orchestration / Kubernetes controller)那一套。**

## 0. 为什么"看起来都一样":原语先于产品

先建立一个心智模型。下面这些概念,**在操作系统、数据库、分布式系统教科书里就存在了**,Redis / Postgres / Kubernetes 只是各自的**实现(implementation)**:

```
        通用原语(理论层)
   ┌────────────┬─────────────┬──────────────┐
   │  互斥锁     │   复制       │   选主        │  ……
   └─────┬──────┴──────┬──────┴───────┬──────┘
         │             │              │
   ┌─────▼─────┐ ┌─────▼─────┐ ┌──────▼──────┐
   │PostgreSQL │ │  Redis    │ │ Kubernetes  │  (各自的展厅/实现)
   │advisory   │ │ SET NX    │ │ Lease 对象  │
   │lock       │ │           │ │             │
   └───────────┘ └───────────┘ └─────────────┘
```

所以当你在 Redis 里手搓 `SET NX` 分布式锁时,你其实是在**应用层重新实现**数据库早就提供的东西。**"这跟数据库一样"不是巧合,是必然** —— 它们在解同一道题。

## 1. 大对照表

| 通用原语 | 本项目(Redis + ARM) | PostgreSQL | 一样吗 |
|---|---|---|---|
| **命名互斥锁**(named mutex) | 分布式锁 `SET lock:key NX PX`(§4.5) | `pg_try_advisory_lock(key)` | 概念一样;**释放机制不同**(见 §3) |
| **悲观锁**(先锁后动) | 锁住整段"查—建—写" | `SELECT ... FOR UPDATE`(行锁) | 一样 |
| **乐观并发控制** | claim + poll(§4.7);或本可用 `WATCH/MULTI/EXEC` | `version` 列 / `SERIALIZABLE` 隔离 | 一样,教科书级(见 §2) |
| **原子的"不存在才写"** | `SET NX` | `INSERT ... ON CONFLICT DO NOTHING` | 一样 |
| **check-then-act 竞态** | 两请求都读到 miss 各建一台 | lost update / phantom | **同一个问题**(见 §4) |
| **复制 · 主从** | Redis primary + replica | streaming replication | 一样,连 failover 丢数据窗口都一样(见 §5) |
| **复制 · 主主** | active-active(仅 Redis Enterprise CRDT) | BDR / Citus / 逻辑复制冲突解决 | **一样地难,难的原因相同** |
| **选主**(leader election) | `SET reaper:leader NX EX` + 续租(§6.1) | `pg_try_advisory_lock` 选单 worker | 一样;和 K8s `Lease` 也一样 |
| **后台回收 / GC** | reaper 扫孤儿 sandbox | `pg_cron` 清过期行 + autovacuum 清死元组 | 概念像,**对象不同**(见 §6) |
| **自动过期 TTL** | Redis 原生 `EXPIRE` | **无原生 TTL**,得自己写清理任务 | Redis 天生带,Postgres 要手搓 |

下面挑几个最值得深挖的,逐个展开。

## 2. 悲观锁 vs 乐观锁 —— 这是纯粹的数据库理论

你用了"乐观锁"这个词,用得**很准**。这对概念是并发控制的核心,和 Redis 无关:

- **悲观锁(pessimistic)**:"我假设一定会冲突,所以**先锁,再动**。" 别人得等我。
- **乐观锁(optimistic)**:"我假设通常不冲突,所以**先动,提交时再检查**有没有人插队;有就重试。" 谁都不等,失败才回头。

三个展厅里各自长这样:

**悲观锁:**
```sql
-- PostgreSQL:锁住这一行,别人 SELECT FOR UPDATE 会阻塞
BEGIN;
SELECT sandbox_id FROM sessions WHERE key = 'oid:sid:group' FOR UPDATE;
-- ……在锁内决策……
COMMIT;  -- 释放
```
```python
# 本项目:锁住整段"查—建—写"(概念等价于 FOR UPDATE)
async with self._dlock(key):          # SET lock:key NX PX
    ...  # 查—建—写
```

**乐观锁:**
```sql
-- PostgreSQL:不锁,靠 version 列在写回时判有没有人动过
UPDATE sessions SET sandbox_id = :new, version = version + 1
WHERE key = :k AND version = :expected;   -- 影响 0 行 = 有人先改了 → 重试
```
```
# Redis 自己也有原生乐观锁:WATCH/MULTI/EXEC
WATCH session:key          # 盯住它
val = GET session:key      # 读
MULTI                       # 开事务
SET session:key <new>       # 想写的
EXEC                        # 若期间 key 被别人改过 → 整个事务作废,返回 nil → 重试
```

§4.7 那个 **claim + poll** 就是乐观思路的变体:不持锁几分钟,只做**一次原子认领**(`SET NX`,等价于 `INSERT ... ON CONFLICT DO NOTHING`),抢到的人去建、没抢到的人轮询等结果。**优点是不长时间阻塞别的副本,缺点是状态机复杂**(认领者中途崩了要超时回收)—— 这也正是数据库里"乐观锁要写重试逻辑"的同款烦恼。

> 记忆钩子:**悲观 = `FOR UPDATE` / 持锁;乐观 = version 列 / `WATCH` / `ON CONFLICT` + 重试。** 本项目默认用悲观(分布式锁),claim+poll 是乐观备选。

## 3. 命名锁:分布式锁 ↔ advisory lock,以及那个致命的释放差异

Redis 的 `SET lock:key NX` 和 Postgres 的 `pg_try_advisory_lock` 是**最像的一对** —— 两者都是**"不绑在任何一行数据上、纯靠一个名字"的锁**,专门把一段临界区在多个连接/客户端之间串起来(区别于 `SELECT FOR UPDATE` 那种绑在具体行上的锁)。

但**释放/故障清理的模型根本不同**,这个差异塑造了 Redis 锁的一切设计:

| | Redis 分布式锁 | Postgres advisory lock |
|---|---|---|
| 谁来"防死锁" | **TTL(`PX`)**:持锁者崩了,锁到期自动消失 | **连接存活**:持锁者的连接一断,服务器**自动**帮它释放 |
| 需要 TTL 吗 | **必须**(Redis 不跟踪"持锁者连接死没死") | **不需要**(session 级锁随连接走,txn 级锁随 `COMMIT`/`ROLLBACK` 走) |
| 需要 token 防误删吗 | **需要**(§4.3:超时后可能删到别人的锁) | 不需要(服务器按 owner 管理) |
| failover 存活吗 | 否(异步复制丢锁,§4.8) | **否,而且更彻底**:advisory lock 是**内存态、不写 WAL**,failover 时**所有** advisory lock 直接蒸发 |

**核心一句:Redis 锁靠"租约(lease)+ 时间",Postgres 锁靠"连接(session)生命周期"。** 这就是为什么 Redis 锁离不开 `PX`、离不开 token、离不开 watchdog 续租(§4.4),而 Postgres 锁这些都不用 —— 服务器替它兜底了。

> 有意思的是:**两者在 failover 时都不安全**(见 §5),因为锁都是**易失(ephemeral)状态**。要跨故障绝对安全,得上 **fencing token**(单调递增号 + 资源方校验),这超出了单个锁原语的能力。

## 4. check-then-act 竞态 ↔ 数据库的隔离级别

§3(那篇)的"两个并发请求都读到 miss,于是都去建一台"——这在数据库里有正式名字:**lost update / phantom read**,是**隔离级别(isolation level)**要解决的经典问题。

数据库给你三种武器:

| 手段 | Postgres 怎么写 | 本项目对应 |
|---|---|---|
| **提高隔离级别** | `SET TRANSACTION ISOLATION LEVEL SERIALIZABLE` —— 冲突时一方报 `40001` 让你重试 | (没直接对应;相当于"让数据库替你检测竞态") |
| **唯一约束兜底** | `INSERT ... ON CONFLICT DO NOTHING` —— 两个并发插入只有一个成功 | `SET session:key NX`(claim,§4.7) |
| **显式锁** | `SELECT ... FOR UPDATE` | 分布式锁(§4.5) |

**如果 sandbox 状态是存在 Postgres 里的一行**,你甚至可以**什么锁都不用**,只靠 `SERIALIZABLE` 或唯一约束,让数据库替你把竞态挡掉,冲突了自动 `ROLLBACK`,干净利落。**本项目做不到这一点** —— 原因就是下一节的 §7②:临界区里那个"建 sandbox"根本不在数据库里,没法回滚。这正是分野的起点。

## 5. 复制:主从 / 主主 —— 连"坑"都一模一样

**主从(primary + 只读 replica)**:两边都是**异步复制**,都有同一个 failover 数据丢失窗口:

```
Redis:    primary 回 OK → 还没复制给 replica → primary 崩 → replica 顶上(缺最后几条)
Postgres: COMMIT 返回   → WAL 还没流到 standby → primary 崩 → standby 顶上(缺最后几条)
```

**解法也一样,取舍也一样**:改**同步复制**(Redis `WAIT` / Postgres `synchronous_commit` + `synchronous_standby_names` 法定人数),用**延迟**换**不丢** —— 天下没有免费的午餐,两边都在这条曲线上选点。

**主主(multi-master)**:两边都**又难又少用**,而且**难的原因完全相同** —— 两个 master 同时写同一个 key,产生**写冲突**,必须有一套裁决机制(last-write-wins / CRDT / 应用层解决):

| | Redis | PostgreSQL |
|---|---|---|
| 原生支持 | ❌ 默认不支持;active-active 靠 **Redis Enterprise CRDT** | ❌ 核心不支持;靠 **BDR / Citus / 逻辑复制**冲突处理 |
| 为什么难 | 并发写冲突无法自动"正确"合并,只能选一种妥协策略 | 同左 |

> 记住:**"主从"到处都能做且便宜;"主主"到处都难且贵** —— 这不是某个产品的短板,是分布式写入的**固有难度**。

## 6. 回收:reaper 到底像什么 —— 从数据库转向编排

reaper 是三个概念里**最不像纯数据库**的一个,拆开看很有意思:

- **像 `pg_cron` 定时清理**:reaper 每 300s 扫一遍删过期的,约等于 `DELETE FROM sessions WHERE expires_at < now()` 挂个定时器。**注意**:Postgres **没有原生 TTL**,过期清理得你自己写;Redis 的 `EXPIRE` 是原生的 —— 所以路由表的 30min 滑动窗口在 Redis 里是一行 `SET ... EX`,在 Postgres 里得靠 cron。
- **像 autovacuum(部分)**:Postgres 的 autovacuum 在后台回收 MVCC 产生的死元组(dead tuples),也是"后台 GC",和 reaper 神似 —— 但清的是**数据库内部的垃圾**。
- **但它真正像的是 Kubernetes controller 的 reconcile 循环** ← 关键。

reaper 干的事,本质是:**"把外部世界的实际状态(ARM 里真实存在的 sandbox),和我 source-of-truth 里记的应该状态(Redis 反向索引),对一遍,多出来的删掉。"** 这正是 K8s controller 的 **level-triggered reconciliation**:

| Kubernetes controller | 本项目 reaper |
|---|---|
| desired state(etcd 里的 spec) | source of truth(Redis 反向索引 `_index`) |
| actual state(集群里真实的 pod) | actual(ARM `list_sandboxes()`) |
| reconcile:多删少补,幂等,反复跑 | reap_orphans:实际多于应该 → 删,幂等,每 300s 跑 |
| leader election 用 `coordination.k8s.io/Lease` | reaper leader 用 `SET reaper:leader NX EX`(§6.1) |

**最后一行尤其漂亮**:K8s 的 `Lease` 对象里就是 `holderIdentity` + `renewTime` + `leaseDurationSeconds` —— **和你 `SET reaper:leader token NX EX` + 续租是一模一样的东西**。选主这件事,在 Redis、Postgres advisory lock、K8s Lease 里是**同一个原语的三种皮肤**。

> 所以 reaper 的定位是:**"数据库不会帮你删掉一台孤儿 VM,因为那不是它的数据。"** reaper 站在数据库和编排的交界上,身子探向编排那边。

## 7. 三点关键分野:它为什么"不只是数据库"

前面全在讲"一样"。现在讲**三处真的不一样** —— 这三点恰恰是这整套 lock + reaper + 幂等 存在的**根因**:

### ① 锁的释放模型:租约 vs 连接
已在 §3 展开。**Redis 靠 TTL/时间,Postgres 靠连接生命周期。** 这决定了 Redis 锁必须自带 token + PX + watchdog,而 Postgres 锁把这些都交给服务器。

### ② 被保护的是"外部、不可回滚的副作用",不是一行数据 ← 最深的一点

数据库事务能 `ROLLBACK`,是因为它保护的是**它自己管的数据**。但本项目临界区里的核心动作 ——`create_sandbox()`—— 是**在 ARM 里开了一台真实的 VM,数据库对它一无所知,更没法回滚**。你没法"回滚一台已经跑了 3 分钟的 sandbox"。

```
数据库世界(ACID):        BEGIN → 改数据 → 冲突?→ ROLLBACK,像什么都没发生
本项目世界(saga):        建 VM → 冲突?→ VM 已经真实存在了,回不去 → 只能"补偿":事后删掉(reaper)
```

**正因为副作用在数据库边界之外,ACID 就失效了**,你被迫退到另一套方法论:

- **尽力而为的锁**(能挡就挡,§4.8 说了它只是"效率锁");
- **幂等(idempotency)**(`_ensure_volume` 吞掉 `409 AlreadyExists`,重复建不出错);
- **事后对账 / 补偿(reaper)**(挡不住的漏网孤儿,事后扫出来删)。

这套东西的正式名字叫 **saga 模式 / 补偿事务(compensating transaction)/ 最终一致性(eventual consistency)**。**一句话:数据库给你 ACID,但 ACID 只覆盖它自己的数据;跨出数据库边界去碰外部资源,你就进入了 saga 的世界。**

### ③ Redis 在这里"就是被当协调数据库用的",可以换成 Postgres

有个有趣的反转:Redis 在这套架构里**扮演的就是数据库角色**(存路由表、当锁、发 TTL)。所以你完全**可以用 Postgres 顶替它**:

| 本项目用 Redis 做的 | 换成 Postgres 怎么做 |
|---|---|
| 分布式锁 `SET NX` | `pg_try_advisory_lock` |
| 路由表 + 30min TTL | 一张 `sessions` 表 + `expires_at` 列 |
| reaper 扫过期 | `pg_cron` 定时 `DELETE WHERE expires_at < now()` |
| reaper 选主 | `pg_try_advisory_lock` 选单 worker |

**都能跑。** 选 Redis 不是因为 Postgres 做不到,而是 Redis **天生为"短命、要 TTL、要快、单次原子 `SET NX`"的协调状态而生**;Postgres 为"持久、关系型、要事务"的数据而生。**这是"选对工具",不是"能力差距"。** 如果这套状态本来就要和业务数据一起放在 Postgres 里,那用 advisory lock 反而更省一个组件。

## 8. 归类:哪些属于"数据库那套",哪些属于"编排那套"

把全部概念拍扁成一张归属表,收束整篇:

| 概念 | 属于哪一套 | 心智锚点 |
|---|---|---|
| 悲观锁 / 乐观锁 | **数据库** | `FOR UPDATE` / version 列 |
| 分布式锁 | **数据库**(协调层) | advisory lock |
| check-then-act 竞态、隔离级别 | **数据库** | lost update / SERIALIZABLE |
| 主从 / 主主复制、failover 窗口 | **数据库**(也是通用分布式) | streaming replication |
| TTL / 过期回收 | 骑墙(Redis 原生 / Postgres 手搓) | `EXPIRE` vs `pg_cron` |
| **reaper 对账孤儿** | **编排** | K8s reconcile loop |
| **选主 leader election** | 骑墙(锁的另一种用法) | Redis Lease = K8s Lease |
| **"建 sandbox 不可回滚" → saga / 补偿** | **编排** | 补偿事务、最终一致性 |

> **本项目的真实形状:一半是"跑在 Redis 上的并发控制"(数据库那套),一半是"协调数据库管不到的外部资源"(编排那套)。** 你熟的 Postgres 知识,能**直接迁移**到左半边;右半边(reaper、幂等、saga)得用 **Kubernetes controller / 云编排**的心智去理解。

---

## 9. 一句话收束

> 你看到的**锁、乐观/悲观、主从/主主、选主**,是**通用的并发控制与分布式系统原语**,数据库只是它们最著名的展厅 —— 所以"像 PostgreSQL"是**真像**,你可以放心地把数据库直觉迁过来。
> 但本项目真正的骨架,是 **"用一个数据存储(Redis)去协调一堆数据库管不到的外部资源(ARM sandbox)"** —— 一旦副作用跨出数据库边界,**ACID 失效,saga 登场**,于是有了 reaper、幂等、尽力而为的锁。**这半边更像 Kubernetes 的 controller,而不是跑在 Postgres 上的 CRUD。**
