# 全新 azd 部署踩坑复盘

> 背景:2026-07-19,在一个**全新订阅**上第一次用 `azd`(拆成 `azd provision` → `azd deploy`)把 ACA 栈端到端部署,并用 MCP 双通道(`diagnose_bash` / `action_bash`)实测。
> 结论先行:**azd 化本身可行,端到端已实测可用**;但全新环境首次部署会连续踩到 5 类坑 —— Graph 扩展、ARM 依赖图、azd hook、Entra 预授权、sandbox 冷启动。本文详解每个坑**为什么**发生、现场怎么修、怎么根治。

---

## 总览

| # | 问题 | 层面 | 根因一句话 | 现状 |
|---|---|---|---|---|
| 1 | `fic` module 找不到 worker SP | Microsoft.Graph 扩展 + ARM preflight | `existing` 引用在 SP 被创建**之前**就被 preflight 校验;`dependsOn` 管不了 preflight | 重跑绕过 · **待根治** |
| 2 | `mcp-app` 循环依赖 | ARM 依赖图(静态) | 同名 `existing` + 主资源 = 自引用环;条件 `if` 不影响静态环检测 | ✅ 已修(拆 `fetch-container-image.bicep`) |
| 3 | OBO secret 没注入进容器 | azd hook 重入 | 在 postprovision hook 内部再调 `azd env set` 不可靠 | 手动补 · **待根治** |
| 4 | 客户端 token 被 server 拒 | Entra 预授权 | MCP API 的 `preAuthorizedApplications` 漏了自建 CLI client | 手动补 · **待根治** |
| 5 | 首次工具调用报错 | sandbox 冷启动 | 冷拉镜像 + 启 microVM > 默认 `SANDBOX_CREATE_TIMEOUT=30s` | 重试即成 · 调大超时 |

**部署时间线**(把坑串起来):
`provision#1` 撞坑①(SP 部分建出)→ `provision#2` 撞坑②(前 6 个资源建出)→ 修坑② → `provision#3` **成功**但坑③(secret 没生效)→ 手动补 secret → `deploy` 成功 → 改 `.mcp.json` 指新部署 + 加组 → 客户端重连撞坑④(token 被拒)→ 修坑④ → 重连成功 → 首次 `diagnose_bash` 撞坑⑤(超时)→ 重试成功、确认新 instance 双通道可用。

## 坑 1 · `fic` module 的 Graph preflight 竞态

**现象**
`azd provision` 首跑失败:
```
InvalidTemplateDeployment: The template deployment 'fic' is not valid ...
NotFound: Request_ResourceNotFound: Resource 'dataops-mcp-diagnose-sp' does not exist ...
```
但事后一查,`dataops-mcp-diagnose-sp` 这个 SP **其实建出来了**。

**为什么(根因)**
- `fic.bicep` 里用 `existing` 引用 worker SP app:
  ```bicep
  resource diagnoseSpApp 'Microsoft.Graph/applications@v1.0' existing = {
    uniqueName: '${name}-diagnose-sp'
  }
  ```
- 对 Microsoft.Graph 扩展资源,解析一个 `existing` 引用 = **去 Graph 按 `uniqueName` 查这个对象**。
- ARM 对每个 module(= 一个 `Microsoft.Resources/deployments`)会做 **preflight 校验**,校验时就会解析其中的 `existing` 引用。这个解析发生在**整包模板的前置校验阶段**,早于 `identity` module 真正**执行**(创建 SP)。
- `main.bicep` 里 `fic` 有 `dependsOn: [identity]`,但 `dependsOn` 只保证**执行顺序**(fic 的资源在 identity 之后 apply),**管不了 preflight 校验的时机**。于是:preflight 时 SP 还不存在 → `fic` 校验失败 → 整包被判 `InvalidTemplateDeployment`。
- **为什么 SP 却建出来了?** Microsoft.Graph 扩展资源不是标准 ARM 资源,它们的写入在部署早期就提交给了 Graph API;整包虽被 `fic` 的校验判失败,但已提交的 Graph 变更**不会回滚**(Graph 无事务)。所以 SP/组留了下来。
- **为什么重跑能过?** SP 是 tenant 级持久对象。第二次跑,`fic` 的 `existing` 在 preflight 就能解析到**已存在**的 SP → 校验通过 → 部署继续。

一句话:**`existing` 跨 module 引用一个"同一次部署里才刚被创建"的 Graph 对象,踩中了 preflight 早于执行的时序**。这是 Microsoft.Graph Bicep 扩展(public preview)的已知模式问题,只有全新环境(对象从没建过)首次部署才暴露。

**现场绕过**:重跑一次 `azd provision`(SP 已存在)。

**根治方向**(择一):
- **A. 结构性(推荐)**:把 FIC 定义**并进 `identity.bicep`**,和 worker SP app 处于同一编译单元 —— 同 module 内引用不走 `existing`,无 preflight 竞态。代价:FIC 需要 sandbox-group MI 的 `principalId`,得让 `identity` 接收它作 param(`main.bicep` 把 `sandboxGroups.outputs` 传进 identity,identity 依赖 sandboxGroups)。
- **B. 两阶段**:先只部署 identity(建 SP),再全量部署。
- **C. 自动化重跑**:preprovision/包装脚本检测首跑失败后自动重跑。治标不治本。

## 坑 2 · `mcp-app` 的循环依赖(本次实现引入的 bug)

**现象**
修完坑①、`provision` 建到第 7 个资源时失败:
```
InvalidTemplate: Circular dependency detected on resource:
'.../containerApps/dataops-mcp-mcp'
```

**为什么(根因)**
- 为了实现"`azd deploy` 换过的真镜像,在重 `provision` 时不被打回占位镜像",我在 `mcp-app.bicep` 里内联了一段镜像回读:
  ```bicep
  resource existingApp '...containerApps@2024-03-01' existing = if (mcpAppExists) {
    name: '${name}-mcp'          // ← 和下面的 app 同名!
  }
  var effectiveImage = mcpAppExists ? existingApp!.properties.template.containers[0].image : mcpImage
  resource app '...containerApps@2024-03-01' = {
    name: '${name}-mcp'          // ← 同名
    ... image: effectiveImage
  }
  ```
- `existingApp` 和 `app` **同名** `${name}-mcp`。`app.image` 依赖 `effectiveImage`,`effectiveImage` 依赖 `existingApp`。在 ARM 的依赖图里,`app` → `existingApp`,而两者指向**同一个资源名** → **`app` 依赖它自己** → 循环。
- 关键:ARM 的循环依赖检测是**静态的**(编译/校验期,看依赖图拓扑),**不看运行时的 `if (mcpAppExists)` 条件值**。所以哪怕首跑 `mcpAppExists=false`,静态图里那条自环依然被判为循环。
- azd 官方 ACA 模板正是为了打破这个环,才把镜像回读放进**独立 module**(`fetch-container-image.bicep`):子 module 是独立编译/部署单元,`existingApp` 在子 module 里,主 `app` 通过 module 的 `output` 拿 image,ARM 把 module 当成一个依赖节点,不产生自环。我实现时图省事内联了,踩坑。

**现场修复(已并入代码)**:新建 `provisioning/aca/modules/fetch-container-image.bicep`,`mcp-app.bicep` 改成调这个 module 拿 `effectiveImage`。`bicep build` 通过、`provision` 通过。

## 坑 3 · postprovision hook 里 secret 注入没生效

**现象**
`provision` 成功后:
- ACR 里**有** `mcp-sandbox` 镜像(hook **第 1 步** `az acr build` 跑了);
- 但 container app 的 `mcp-client-secret` **还是占位符** `placeholder-set-via-secret-set`,且 azd env 里**没有** `MCP_CLIENT_SECRET`(hook **第 2 步**没生效)。

**为什么(根因,最可能推断)**
hook `postprovision.sh` 第 2 步是:
```sh
if [ -z "$(azd env get-value MCP_CLIENT_SECRET ...)" ]; then
  secret=$(az ad app credential reset --id "$MCP_APP_ID" ...)
  azd env set MCP_CLIENT_SECRET "$secret"          # ← 嫌疑点
  az containerapp secret set ... mcp-client-secret="$secret"
fi
```
- 第 1 步能跑(镜像在)证明 hook **确实执行了**、azd 注入的环境变量(如 `REGISTRY_NAME`)也在。
- 最可能根因:**在 azd hook 内部再调用 `azd`(`azd env get-value` / `azd env set`)不可靠**。azd 跑 postprovision hook 时,父 azd 进程正持有当前 environment 的上下文;hook 里再起一个 `azd` 子进程去读写**同一个** env,会出现读不到 / 写入被父进程后续状态覆盖 / 子进程找不到 environment 等问题(azd 对 hook 内重入自身没有可靠保证)。
- 结果:`azd env set` 没落盘,`az containerapp secret set` 那步也可能因为 `set -eu` 在前面 `azd env set` 异常时被跳过(或整段没跑完),secret 停在占位符。
- 佐证:`provision` 报 `SUCCESS`,若 hook 以非零退出,azd 应报 hook 失败 —— 说明 hook"看起来成功"了,但 secret 副作用没落地,符合"azd 重入静默失灵"的画像。

**现场绕过**:手动跑 hook 第 2 步该做的 —— `az ad app credential reset` + `az containerapp secret set` + `azd env set`(在 hook **外面**,单独一个 shell,不重入 azd)。验证 container secret 变成真值。

**根治方向**:让 secret 注入**不依赖 hook 内的 `azd env` 重入**:
- 幂等 guard 从"查 azd env"改成"查 container app secret 是否为占位符"(用 `az containerapp secret show`);
- 或者把 `azd env set` 那步去掉,secret 仅由 hook 直接 `az containerapp secret set` 注入(接受"每次首部署 reset 一次、值不进 azd env、重 provision 由 Bicep 占位再被 hook 覆盖"的取舍);
- 需要针对 azd 版本实测哪种可靠(本次未验证 azd hook 重入的确切行为)。

## 坑 4 · MCP API 的 `preAuthorizedApplications` 漏了自建 CLI client

**现象**
改 `.mcp.json` 指向新部署、客户端重新认证:**能拿到 token(登录成功),但 server 在 reconnect 时拒绝**(`POST /mcpproxy 401`),工具连不上。

**为什么(根因)**
- 查 MCP API app 的授权配置:
  ```
  identifierUris:   ["api://dataops-mcp-mcp-server"]   ✓
  preAuthApps:      ["aebc6443-..."]                    ← 只有 VS Code 内置 client
  scopes:           ["user_impersonation"]              ✓
  ```
  **只 pre-authorize 了 VS Code(`aebc6443`),没有自建的 CLI client(`32b136f2`)。**
- OAuth 有**两道独立的关**:
  1. **拿 token**:客户端用 CLI client `32b136f2` 走 PKCE,Entra 认证用户、发一个 token。这一步和"该 client 有没有被 API 预授权"基本无关 —— 能登录就发得出 token(客户端因此报"认证成功")。
  2. **token 对该 API 有效吗**:server 验 token 的 `aud`/`scope`(要求 `aud = MCP API`、含 `user_impersonation`)。
- `preAuthorizedApplications` 管的是**第二关**:让指定 client **无需额外 consent** 就能拿到针对该 API 的正确 scope、且 token `aud` 指向该 API。`32b136f2` 不在名单里 → Entra 发的 token 对 MCP API 无效(`aud`/`scope` 不满足)→ server 401。
- 根因在 `identity.bicep`:它建了 CLI client,却没把这个 CLI client 加进 MCP API 的 `preAuthorizedApplications`(只显式列了 VS Code)。这也和 README 提过的"Graph 扩展更新 `preAuthorizedApplications` 不稳"同源。
- **为什么"能认证成功"却连不上**:因为"拿到 token" ≠ "token 被 API 接受"。客户端只知道自己拿到了凭据,不知道那凭据对 server 无效 —— 最终裁判永远是 server 的 JWT 校验。

**现场修复**:`az rest PATCH` 把 CLI client `32b136f2` 加进 MCP API 的 `preAuthorizedApplications`(保留 VS Code),带 `user_impersonation` 的 scope id。等 Entra 传播 1–2 分钟后,客户端重新认证即通过。

**根治方向**:`identity.bicep` 的 `preAuthorizedApplications` 里**加上 `cliClientAppId`**(和 VS Code 并列)。

## 坑 5 · sandbox 冷创建超时(含 timeout 记录)

**现象**
新部署后**首次**调 `diagnose_bash` 直接报错(空错误);**立刻重试第二次就成功**,返回正确的 SP 身份。

**为什么(根因)**
- 首次调用会触发 `SandboxManager` 为当前 Session/group **冷创建**一个 ACA sandbox(microVM),链路包括:调 sandboxGroups API 创建 sandbox → 从 `SANDBOX_DISK_IMAGE`(ACR 的 `mcp-sandbox:latest`)**拉镜像** → 启动 microVM → FIC 无密码 `az login` bootstrap → 挂载 Blob workspace。
- **第一次**:该镜像还没被这个 ACA 环境缓存,"拉镜像 + 冷启 microVM"的总耗时**超过了默认的 `SANDBOX_CREATE_TIMEOUT = 30s`** → 创建超时 → 工具报错。
- **重试**:镜像/microVM 已就绪或被缓存,创建很快 → 成功。
- 这不是逻辑 bug,而是**冷启动延迟 vs 超时阈值**的配置问题:默认 30s 对"首次冷拉镜像"偏紧。

**现场绕过**:重试一次即可(镜像缓存后就快了)。

**根治 / 调优**:
- 调大 `SANDBOX_CREATE_TIMEOUT`(如 60–90s)。**这个值恰好是本次 azd 化后新增的、`azd env set` 可控的调优 knob**:
  ```bash
  azd env set SANDBOX_CREATE_TIMEOUT 90 && azd provision
  ```
- 或预热:部署后先"空跑"一次触发镜像拉取,让后续用户走热路径。

## 附:一次干净的全新部署应该怎么跑(workaround 清单)

在坑 1/3/4 根治**之前**,全新环境按下面顺序可稳定跑通:

1. `azd env new <name> --subscription <sub> --location westus2`
2. `azd provision` —— **预期首跑因坑① 失败**(建出 SP/组)。
3. `azd provision` —— 第二次(SP 已存在)通过 provision(坑② 已在代码修复,不会再犯)。
4. **手动补 secret**(坑③):
   `az ad app credential reset` → `az containerapp secret set mcp-client-secret=...` → `azd env set MCP_CLIENT_SECRET ...`
5. `azd deploy` —— 构建 + 换 mcp-server 镜像。
6. **手动补 CLI client 预授权**(坑④):把 `CLI_CLIENT_APP_ID` 加进 MCP API 的 `preAuthorizedApplications`。
7. 把用户加进 diagnose / action AD 组;客户端指向 `https://<MCP_FQDN>/mcpproxy`。
8. 首次工具调用若超时(坑⑤)**重试一次**;或先 `azd env set SANDBOX_CREATE_TIMEOUT 90`。

根治坑 1/3/4 后,理想目标是 `azd up` + 加组 + 指客户端即可,无需 2–4、6 的手动步骤。
